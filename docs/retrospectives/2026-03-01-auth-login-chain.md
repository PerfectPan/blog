# 2026-03-01 登录链路修复复盘

## 背景

在 `TanStack Start + Better Auth + Payload` 迁移后，登录链路出现过两类问题：

1. `POST /api/auth/sign-in/email` 在生产返回 `500`
2. GitHub 登录回跳后，头部仍显示 `Login / Sign Up`，看起来像“没登录”

本文记录「第一次为什么没改对」和本次修正后的约束。

---

## 第一次没改对：原因

### 1) 技术判断失误

- 我第一次把“头部登录态展示”主要放在服务端 loader 注入方案上，假设回跳后会稳定刷新。
- 实际上这个点在当前实现里不稳：回跳时机 + loader 刷新策略没有形成可靠闭环，导致 UI 不一致。

### 2) 验证范围不完整

- 当时验证了接口可用性，但没有把“GitHub OAuth 完整回跳 + 头部状态变化 + 认证页重定向”作为一个完整冒烟链路跑完。
- 结果是接口层面看起来正常，但用户可见的登录态仍不正确。

### 3) 交付节奏问题

- 先做了结构调整，再补齐链路验证，顺序反了。
- 对登录这种跨页面状态问题，应该先固定验收场景，再改实现。

---

## 这次怎么修

### 已落地修复

- `Header` 改为消费 `authClient.useSession()`，直接以真实会话渲染登录态。
- `/login`、`/signup` 页面增加会话检测，已登录时主动跳转 `/blog`。
- 认证路由在处理前保证 Better Auth schema 已就绪，避免首次请求撞到缺表导致 500。

### 不变约束（继续保留）

- 博客访问权限裁决仍在服务端（`/blog`、`/blog/:slug` 的最终可见性不在前端判定）。
- Header 只负责“状态展示”，不承担权限裁决。

---

## 防再发措施

后续涉及登录链路改动，必须最少通过以下检查：

1. 游客访问 `/blog`，头部显示 `Login / Sign Up`
2. Email 登录成功后，头部显示 `Logout`（且不再显示 `Login / Sign Up`）
3. GitHub 登录回跳后，头部状态正确
4. 已登录访问 `/login`、`/signup` 会跳转 `/blog`
5. 登出后刷新页面，头部恢复游客态

部署前额外检查：

1. 已执行 Better Auth 迁移（`pnpm migrate:auth`）
2. 已执行 `pnpm --filter @blog/web typecheck`
3. 已执行 `pnpm --filter @blog/cms typecheck`

---

## 结论

这次问题本质不是“单点代码错误”，而是**登录态展示链路缺少端到端验收闭环**。  
修复后已经把 UI 会话感知与服务端权限边界分开，并补上了最小冒烟清单，后续按清单执行即可显著降低复发概率。
